# MiniTodo — Vision

> **产品一句话**：一个极简的待办列表应用——记录、查看、完成，仅此而已。

> **北极星**：让用户用最低的认知负担记录"接下来要做什么"。（引用 PRODUCT_PHILOSOPHY.md 北极星，一致）

## Problem Space

用户在日常生活中会不断产生"接下来要做什么"的念头——买牛奶、回邮件、修水龙头。这些念头如果留在脑子里，会占用工作记忆，导致焦虑和遗忘。

现有的待办应用要么太重（OmniFocus、Todoist 有标签/优先级/项目/子任务），要么太轻（系统备忘录没有持久化保障）。用户需要的不是"生产力系统"，是一个**可信的、零摩擦的、认知卸载入口**。

---

## Intent List

### INT-001: User Authentication

**意图叙事**：

用户需要自主管理自己的身份——这是产品信任的基础。如果用户不能控制自己的身份，待办列表就只是一个匿名记事本，数据归属感无从谈起。认证不是安全措施，是身份自治的入口，是"这是我的列表"的确认。

如果砍掉认证，用户无法区分"我的待办"和"别人的待办"，数据归属感（PRODUCT_PHILOSOPHY.md#non-negotiable-values 第 2 条）就失去了根基。认证是信任的第一步——用户信任这个列表，首先需要知道这个列表是他的。

**验收契约**：

- 用户可以通过用户名+密码注册新账户
- 注册后可以通过用户名+密码登录
- 登录状态在会话内持久化（刷新页面不丢失）
- 登录后用户只能看到自己的待办项
- 错误的凭证产生明确的错误提示

**哲学锚点**：
- `PRODUCT_PHILOSOPHY.md#non-negotiable-values` — 数据归属感
- `ENGINEERING_CREED.md#anti-patterns` — 禁止硬编码（密码处理）

---

### INT-002: Create Todo

**意图叙事**：

记录是认知卸载的核心动作。用户脑子里冒出一个念头，需要在三秒内把它从脑子搬到列表里。这个动作的摩擦决定了整个产品的成败——多一个步骤，用户就会选择"算了，我记在脑子里"，然后遗忘，然后不信任这个列表。

如果砍掉这个意图，产品就退化成"只能看不能写的列表"，认知卸载完全失败。这个意图服务的北极星是"最低认知负担记录"——创建待办就是那个"记录"动作本身。

**验收契约**：

- 登录用户可以在输入框中输入待办内容并提交
- 提交后待办项立即出现在列表中（无需刷新）
- 待办项有唯一标识、内容、创建时间、完成状态
- 空内容不允许提交（有明确提示）
- 待办内容持久化到本地存储，重启后不丢失

**哲学锚点**：
- `PRODUCT_PHILOSOPHY.md#core-belief` — 认知卸载工具
- `PRODUCT_PHILOSOPHY.md#non-negotiable-values` — 记录零摩擦
- `ENGINEERING_CREED.md#anti-patterns` — 禁止过度抽象

---

### INT-003: List Display

**意图叙事**：

列表是用户和产品之间的界面。用户打开应用的第一眼看到的就是列表——它必须完整、清晰、不加修饰。可见即全部（PRODUCT_PHILOSOPHY.md#non-negotiable-values 第 3 条）不是 UI 偏好，是信任契约。如果列表隐藏了什么，用户就会怀疑"是不是漏了"，信任就崩了。

如果砍掉这个意图，用户创建了待办却看不到，产品就变成了一个黑洞——输入有响应，输出没有。认知卸载需要"确认已记录"的反馈循环，列表展示就是这个反馈。

**验收契约**：

- 登录用户可以看到自己所有未完成的待办项
- 待办项按创建时间倒序排列（最新的在最上面）
- 每个待办项显示内容、创建时间
- 待办项可以标记为完成，完成后视觉降级（灰色/删除线）但仍可见
- 列表为空时显示空状态提示
- 已完成的待办项不隐藏，视觉降级但不消失

**哲学锚点**：
- `PRODUCT_PHILOSOPHY.md#non-negotiable-values` — 可见即全部
- `PRODUCT_PHILOSOPHY.md#anti-patterns` — 不做智能排序
- `DECISION_RUBRIC.md#conflict-resolution-rules` — 可见 > 整洁

## Not Doing

- **不做注册邮箱验证** — 摩擦太大，MVP 阶段用户名+密码足够。理由：认证是身份确认，不是安全堡垒。
- **不做密码找回** — 本地存储的 MVP 不需要。理由：增加复杂度但不服务认知卸载使命。
- **不做待办编辑** — 创建后不可改内容。理由：编辑是"管理"行为，不是"记录"行为。要改就删了重建。
- **不做批量操作** — 不做全选/批量删除。理由：增加 UI 复杂度，不服务零摩擦使命。

## Intent Narrative Guide

意图叙事不是功能描述。区别：

**坏**（功能描述）："实现用户登录功能，支持用户名和密码。"
→ 这只说了"做什么"，没说"为什么做"。

**好**（意图叙事）："用户需要自主管理自己的身份——这是产品信任的基础。如果用户不能控制自己的身份，产品就只是一个旁观者，不是参与者。"
→ 这说了"为什么存在"、"服务什么北极星"、"砍掉会损失什么"。

意图叙事是 Keeper 验证的依据——Keeper 对照叙事判断"实现是否忠实于意图"。叙事越清晰，验证越精确。
